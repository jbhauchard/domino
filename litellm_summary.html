LiteLLM & temperature=0.0

Non-determinism: causes, diagnosis, and evidence

1. Is LiteLLM silently changing 0.0 → 0.1?

Unlikely, but easy to rule out. Enable verbose logging to inspect the exact payload sent to the provider:

import litellm
litellm.set_verbose = True
response = litellm.completion(model="...", messages=[...], temperature=0.0)

The log shows the raw JSON. Confirm temperature is 0.0. You can also check LiteLLM's source (litellm/main.py) for any temperature transformation logic.

2. More Likely Culprits

  • Server-side non-determinism (most likely). Providers like OpenAI document that even at temperature=0, outputs are not guaranteed deterministic due to GPU cluster parallelism. A stable A/B pattern is a classic symptom of two distinct backends serving your requests.
  • Different model versions behind an alias. Aliases like gpt-4 or claude-3-5-sonnet may silently route to different underlying checkpoints depending on provider load or rollout strategy.
  • Prompt variability. Any injected timestamps, session IDs, or dynamic context will shift the output even at temperature=0.

3. How to Build Evidence for FP Non-Determinism

Step 1 — Characterise the distribution.
Run the same fixed prompt ~50 times and count outcomes. True floating-point noise produces rare, random divergences. A stable bimodal split (always A or B, never C) strongly suggests two distinct backends, not FP noise.

Step 2 — Check system_fingerprint (the smoking gun).

print(response.system_fingerprint)  # OpenAI includes this per response
print(response._hidden_params)       # LiteLLM metadata

OpenAI's system_fingerprint changes when server configuration changes. If two distinct fingerprints correlate perfectly with answers A and B, the cause is backend routing — not floating-point arithmetic.

Step 3 — Isolate with a local model.
Run the same model locally (e.g. via llama.cpp or vllm) with a fixed seed and temperature=0. If you get 100% answer A locally but A/B from the cloud provider, the non-determinism is definitively server-side infrastructure.

Bottom line: check response.system_fingerprint. A stable A/B pattern almost always means you are being routed to two different backend servers — not floating-point non-determinism.
